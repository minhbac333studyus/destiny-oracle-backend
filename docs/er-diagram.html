Relationships & Cascade Rules — what happens when a parent row is deleted

Parent Table → Child Table FK Column Cardinality JPA Cascade DB Cascade Note
users → destiny_cards user_id 1 : N NONE none Delete user manually; cards are not auto-deleted
destiny_cards → card_stage_content card_id 1 : N (max 6) PERSIST + MERGE @OnDelete CASCADE Stage history preserved; DB cascade removes rows when card is deleted
destiny_cards → card_images card_id 1 : N (max 6) PERSIST + MERGE @OnDelete CASCADE Image audit trail preserved; DB cascade on explicit card delete
destiny_cards → habits card_id 1 : N ALL + orphanRemoval none Habits fully owned by card; removing from list deletes them
destiny_cards → generation_jobs card_id 1 : N ALL + orphanRemoval @OnDelete CASCADE Jobs and all their steps deleted when card is removed
generation_jobs → generation_job_steps job_id 1 : 12 ALL + orphanRemoval @OnDelete CASCADE 12 steps (6 PROMPT + 6 IMAGE) pre-created as WAITING; each saved via REQUIRES_NEW
habits → habit_completions habit_id 1 : N NONE none Delete completions explicitly before removing a habit
goals → milestones goal_id 1 : N ALL + orphanRemoval none Milestones fully owned by goal
users → goals user_id 1 : N logical only no FK UUID stored as plain column, no DB constraint
users → parsed_activities user_id 1 : N logical only no FK UUID stored as plain column, no DB constraint
users → habit_completions user_id 1 : N logical only no FK Denormalized for quick query; no DB constraint

PromptStatus — destiny_cards.prompt_status — tracks Step 2 (image prompt generation)

NONE GENERATING READY FAILED
NONE → prompts not yet generated · GENERATING → Claude running · READY → prompts exist, Gemini can start · FAILED → Step 2 errored
Regenerating stage content resets status to NONE so image prompts also regenerate.

JobStatus — generation_jobs.status — overall pipeline run state

QUEUED PROMPTING IMAGING COMPLETED FAILED
QUEUED → job created, not started · PROMPTING → Claude writing prompts (steps 0–5) · IMAGING → Gemini generating images (steps 6–11) · COMPLETED → all done · FAILED → pipeline errored; check error_message

StepStatus — generation_job_steps.status — individual step lifecycle

WAITING RUNNING DONE SKIPPED FAILED
All 12 steps start as WAITING — UI sees full pipeline immediately before execution begins.
SKIPPED = prompt already in DB (cost saving — Claude not called again). Gate: IMAGE steps never start until all PROMPT steps are DONE or SKIPPED.
Each status change commits in its own REQUIRES_NEW transaction so UI polling always sees current state.

JobPhase — generation_job_steps.phase — which AI is running

PROMPT IMAGE
PROMPT = steps 0–5 (Claude writes image prompt per stage, sequential) · IMAGE = steps 6–11 (Gemini Imagen generates PNG, all 6 fire in parallel via virtual threads)

Valid values for habits.frequency: "daily" | "weekly"

Valid values for goals.status: "active" | "completed" | "paused"

Valid values for milestones.status: "pending" | "achieved"

Developer Notes

🔑 Authentication

  • No login system — pass X-User-Id header (UUID)
  • Missing header → falls back to default demo user
  • Default user UUID: 00000000-0000-0000-0000-000000000001
  • Demo user seeded by DataInitializer.java on first boot

🌱 Seeded Data (on first boot)

  • 1 demo user (default UUID above)
  • 5 starter cards: health, career, finances, relationships, family
  • 2 sample goals with milestones
  • Remaining 5 aspects added via POST /api/v1/cards

🤖 AI Pipeline (3-step, mandatory order)

  • Step 1 (Claude): fear+dream → title/tagline/lore × 6 stages
  • Step 2 (Claude): title+lore → 6 Imagen-ready text prompts
  • Step 3 (Gemini): 6 prompts → 6 PNG images in parallel
  • Without API keys: Steps 1+2 fallback; Step 3 returns placeholder URLs

⚙️ Job Tracking (UI polling)

  • 12 steps pre-created as WAITING before pipeline starts
  • Step saves use REQUIRES_NEW — visible immediately to UI
  • Poll: GET /{cardId}/jobs/latest every 2–3 seconds
  • Stop polling when job.status = COMPLETED | FAILED

⚙️ Required Environment Variables

  • ANTHROPIC_API_KEY — Claude API key (Steps 1+2)
  • GOOGLE_CLOUD_PROJECT_ID — GCP project for Gemini
  • GCS_BUCKET — Google Cloud Storage bucket
  • DATABASE_URL / USERNAME / PASSWORD — optional (defaults to localhost)

🗃️ Schema Management

  • ddl-auto: update — Hibernate auto-creates tables
  • No Flyway migrations needed for development
  • All IDs are UUID v4 (PostgreSQL uuid type)
  • All timestamps use Instant → stored as timestamptz

🌐 API Base URL

  • Backend: http://localhost:8080
  • Landing page: /
  • Swagger UI: /swagger-ui.html
  • Admin Console: /docs/admin-test.html
  • Pipeline logs: tail -f logs/pipeline.log

📂 Log Files

  • logs/destiny-oracle.log — all application logs (30 days)
  • logs/pipeline.log — AI pipeline steps only (14 days)
  • Both roll daily; live view: tail -f logs/pipeline.log
  • Pipeline loggers at DEBUG; root at INFO
